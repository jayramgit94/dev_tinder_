## Core RESTful Concepts

### What is REST?
REST (Representational State Transfer) is an architectural style for building APIs. It uses standard HTTP methods and URLs to represent resources and operations.

**Key Principles:**
- **Resources**: Data entities (users, posts, etc.)
- **URIs**: Unique identifiers for resources (e.g., `/users`, `/user/123`)
- **HTTP Methods**: Actions on resources (GET, POST, PATCH, DELETE)
- **Stateless**: Each request contains all information needed
- **Standard Status Codes**: 200 (OK), 201 (Created), 404 (Not Found), 500 (Error)

**Example:**
```
GET /users       → Fetch all users (read)
POST /signup     → Create a new user (write)
PATCH /user/123  → Update user with id 123 (modify)
DELETE /user/123 → Delete user with id 123 (remove)
```

## HTTP Methods

### 1. GET - Retrieve Data (Safe & Idempotent)
- **Purpose**: Fetch data without modifying anything
- **Idempotent**: Multiple identical requests return the same result
- **Body**: No request body (data in URL/query parameters)
- **Response**: 200 OK with data, or 404 if not found

### 2. POST - Create New Data
- **Purpose**: Create a new resource
- **Non-Idempotent**: Multiple identical requests create multiple resources
- **Body**: Contains the data to save (JSON)
- **Response**: 201 Created with the created resource, or 400 Bad Request

### 3. PATCH - Partial Update
- **Purpose**: Update specific fields of an existing resource
- **Idempotent**: Updating the same way multiple times has same effect
- **Body**: Contains only the fields to update (JSON)
- **Response**: 200 OK with updated resource, or 404 if not found

### 4. PUT - Complete Replacement
- **Purpose**: Replace entire resource (not commonly used in modern APIs)
- **Body**: Complete resource object
- **Response**: 200 OK or 201 Created

### 5. DELETE - Remove Data
- **Purpose**: Delete a resource
- **Response**: 204 No Content, or 404 if not found

### Route 3: PATCH /user/:id - Update User
**Location**: [src/app.js](src/app.js#L103-L118)

```javascript
app.patch("/user/:id", async (req, res) => {
  try {
    const updatedUser = await User.findByIdAndUpdate(req.params.id, req.body, {
      new: true,
      runValidators: true,
    });

    if (!updatedUser) {
      return res.status(404).send("user not found");
    }

    res.status(200).send(updatedUser);
  } catch (err) {
    console.log("error in updating the user in database", err);
    res.status(500).send("Internal Server Error");
  }
});
```

**What it does**: Updates specific fields of a user by MongoDB id

**Postman Setup:**
- Method: PATCH
- URL: http://localhost:3000/user/662d1234abcd5678ef901234 (replace with real _id from signup)
- Headers: Content-Type: application/json
- Body (raw JSON - only fields to update):
```json
{
  "city": "Pune",
  "about": "Now living in Pune!",
  "age": 26
}
```

**Response (200 OK):**
```json
{
  "_id": "662d1234abcd5678ef901234",
  "firstName": "Jayram",
  "lastName": "Kumar",
  "email": "jayram@example.com",
  "password": "secure123",
  "age": 26,
  "gender": "male",
  "city": "Pune",
  "photoUrl": "https://cdn-icons-png.flaticon.com/512/149/149071.png",
  "about": "Now living in Pune!",
  "skills": ["Node.js", "MongoDB"],
  "__v": 0
}
```

**Important Concepts:**
- `:id` is a URL parameter (extracted from the path)
- `req.params.id` accesses this value
- `new: true` returns the updated document (not the old one)
- `runValidators: true` checks schema constraints
- Only sends fields you want to update (not the entire object)

**How to get the id:**
1. Run POST /signup first
2. Copy the `_id` from the response
3. Use it in PATCH URL

### MongoDB Operations Used

#### 1. Save (Create)
```javascript
const user = new User(req.body);
await user.save();
```
- Creates a new document
- Validates against schema
- Returns the saved document with `_id`

#### 2. FindOne (Read Single)
```javascript
const user = await User.findOne({email: email});
```
- Returns first document matching the query
- Returns `null` if not found
- Common operators: `{email: "test@example.com"}`, `{_id: ObjectId}`

#### 3. Find (Read Multiple)
```javascript
const users = await User.find({});
```
- Returns array of documents matching query
- `{}` means no filter (returns all)
- Returns `[]` if no matches found

#### 4. FindByIdAndUpdate (Update)
```javascript
const updated = await User.findByIdAndUpdate(id, update, {new: true, runValidators: true});
```
- Finds by `_id` and updates
- `new: true` → returns updated document
- `runValidators: true` → validates updated data
- Returns `null` if id not found

#### 5. FindByIdAndDelete (Delete)
```javascript
const deleted = await User.findByIdAndDelete(id);
```
- Finds by `_id` and deletes
- Returns the deleted document
- Returns `null` if id not found

### Common Postman Issues & Solutions

| Issue | Solution |
|-------|----------|
| 404 Not Found | Check URL spelling, HTTP method, and that server is running |
| "Cannot POST /signup" | Server not running or route not defined |
| Empty body response | Forgot to send `Content-Type: application/json` header |
| "user not found" on PATCH | Wrong MongoDB `_id` format or ID doesn't exist |
| "E11000 duplicate key error" | Email already exists (unique constraint) |
| Cannot read property 'email' | Forgot to send request body or body format wrong |

## Short Revision Notes

- `POST` creates data.
- `GET` reads data.
- `PATCH` updates partial data.
- `req.body` for request data.
- `req.params` for URL path values.
- `req.query` for query string values.
- `express.json()` is required for JSON requests.
- Mongoose schema controls type, required fields, and defaults.
- MongoDB `_id` is the best identifier for update/delete operations.
- Good APIs return clear status codes and meaningful error messages.

## Best Practices

### 1. URL Design (REST Standards)

**Good Examples:**
```
GET    /users           → Fetch all users
GET    /users/123       → Fetch user with id 123
POST   /users           → Create new user
PATCH  /users/123       → Update user 123
DELETE /users/123       → Delete user 123

GET    /feed            → Fetch all posts (feed)
POST   /feed            → Create post
```

**Bad Examples:**
```
GET    /getUsers        → Method already in GET, redundant
POST   /createUser      → Method already in POST, redundant
GET    /user-list       → Use plural consistently
PATCH  /update-user/123 → Method already in PATCH
```

### 2. HTTP Status Codes

```
200 OK              → Request succeeded
201 Created         → New resource created
204 No Content      → Success but no response body (DELETE)
400 Bad Request     → Client sent invalid data
401 Unauthorized    → Authentication failed
403 Forbidden       → Authentication passed, but permission denied
404 Not Found       → Resource doesn't exist
500 Server Error    → Server-side error
503 Service Error   → Server temporarily unavailable
```

### 3. Error Handling Pattern

```javascript
app.post("/signup", async (req, res) => {
  try {
    // Validate input
    if (!req.body.email) {
      return res.status(400).send("Email is required");
    }

    // Attempt operation
    const user = new User(req.body);
    await user.save();

    // Success response with correct status code
    res.status(201).send(user);

  } catch (err) {
    // Log error for debugging
    console.log("Error in signup:", err);

    // Send appropriate error response
    if (err.code === 11000) {
      res.status(400).send("Email already exists");
    } else {
      res.status(500).send("Internal Server Error");
    }
  }
});
```

### 4. Request/Response Data Flow

```
Postman
   ↓
express.json() middleware
(converts JSON string → JavaScript object)
   ↓
req.body = { firstName: "Jayram", email: "..." }
   ↓
Route handler logic
   ↓
res.status(201).send(user)
   ↓
Postman (receives response)
```

### 5. Naming Conventions

```javascript
// ✅ Good
app.post("/signup", async (req, res) => {
  const user = new User(req.body);
  await user.save();
})

// ❌ Bad
app.post("/signup", async (req, res) => {
  const u = new User(req.body);
  await u.save();
})
```

### 6. Async/Await Pattern

```javascript
// ✅ Always use async/await for database operations
app.get("/users", async (req, res) => {
  try {
    const user = await User.findOne({email: req.body.email});
    res.status(200).send(user);
  } catch (err) {
    res.status(500).send("Error");
  }
});

// ❌ Don't use Promise then/catch (old pattern)
app.get("/users", (req, res) => {
  User.findOne({email: req.body.email}).then(...).catch(...)
});
```

# Interview Questions and Answers

### 1. What is the difference between REST API and traditional web application routing?
REST API routing is resource-oriented. You design URLs around entities like users, posts, and comments, and use HTTP methods to define the action. Traditional web routing often serves pages or views directly. In REST, `GET /users` means fetch users, while in a web app it may render a users page.

### 2. Why do we use `express.json()` in an Express app?
`express.json()` parses incoming JSON request bodies and converts them into JavaScript objects. Without it, `req.body` will usually be `undefined` for JSON requests, and Postman data will not be available in the route handler.

### 3. What is the purpose of `req.body`, `req.params`, and `req.query`?
`req.body` contains data sent in the request body, usually for POST, PATCH, or PUT. `req.params` contains route parameters like `:id` from the URL path. `req.query` contains query string values like `?email=test@example.com`. A strong API designer chooses the right one based on the use case.

### 4. When should you use POST vs PATCH?
Use POST when you want to create a new resource. Use PATCH when you want to update only some fields of an existing resource. For example, `POST /signup` creates a user, while `PATCH /user/:id` updates only `city` or `about`.

### 5. What is the difference between PUT and PATCH?
PUT usually replaces the full resource, while PATCH updates only selected fields. If a user document already has 10 fields and you send 2 fields, PATCH is the correct choice. PUT is more strict and is often used when the client sends the complete object.

### 6. Why is `new: true` used in `findByIdAndUpdate`?
By default, Mongoose returns the old document before the update. `new: true` tells Mongoose to return the updated document, which is more useful for API responses because the client immediately sees the latest saved state.

### 7. Why do we use `runValidators: true` in update operations?
Without this option, Mongoose may skip schema validation during updates. With `runValidators: true`, fields like `email`, `firstName`, and `password` still follow schema rules when you patch a document.

### 8. What happens if a required field is missing during signup?
Mongoose throws a validation error because the schema requires fields like `firstName`, `email`, and `password`. In a production API, you should catch that error and return a clear 400-level response instead of only sending a generic 500.

### 9. Why is `email` marked as `unique: true`?
Because each user should have one unique identity in the system. If the same email is allowed twice, login, password recovery, and account ownership become ambiguous. `unique: true` also helps prevent duplicate records.

### 10. What does `User.find({})` do?
It fetches all documents from the users collection. The empty object means no filter is applied. In your app, this powers the `/feed` route.

### 11. What is the difference between `findOne` and `findById`?
`findOne` searches by any field or combination of fields, such as `{ email: "test@example.com" }`. `findById` is a shortcut for searching by MongoDB `_id`. If you already have the id, `findById` is cleaner.

### 12. Why should GET requests usually not use a request body?
GET is meant for retrieval, and request bodies are not consistently supported or used with GET. Query parameters are the standard choice for filters and lookup values. A more RESTful design for your route would be `GET /users?email=test@example.com`.

### 13. What is a status code and why does it matter?
Status codes tell the client what happened. `201` means resource created, `200` means success, `404` means not found, and `500` means server-side failure. Good APIs always send meaningful status codes so the frontend or Postman user can understand the result.

### 14. Why is `PATCH /user/:id` better than `PATCH /user`?
Because the URL clearly identifies the specific resource being updated. The `:id` makes the API explicit and scalable. Without the id, the server does not know which user document to update.

### 15. What is MongoDB `_id` and why do we use it?
`_id` is the primary identifier for every MongoDB document. It is automatically generated by MongoDB and is unique. You use it for lookup, update, and delete operations because it is guaranteed to identify a single record.

### 16. What is the practical meaning of idempotency?
An idempotent request gives the same final result even if it is repeated many times. `PATCH` can be idempotent if the same patch is applied repeatedly with the same values. `POST` is usually not idempotent because repeating it may create multiple records.

### 17. How would you explain the request lifecycle in your app?
First, Postman sends the request to Express. Then middleware like `express.json()` parses the body. Next, the route handler reads `req.body`, `req.params`, or `req.query`. After that, Mongoose talks to MongoDB. Finally, the API sends a response with a status code and data back to Postman.

### 18. Why is error handling important in API routes?
Without error handling, the server can crash or return confusing responses. A good API catches database and validation errors, logs them for debugging, and returns a clear message to the client. This makes the system reliable and easier to maintain.

### 19. How would you design a better user API than the current one?
A cleaner design would be:
`POST /users` to create a user,
`GET /users` to fetch all users,
`GET /users/:id` to fetch one user,
`PATCH /users/:id` to update,
`DELETE /users/:id` to delete.
This is more consistent and closer to standard REST conventions.

### 20. What is the main difference between frontend validation and backend validation?
Frontend validation improves user experience by catching mistakes early, but it can be bypassed. Backend validation is mandatory because it protects the database from bad or malicious data. In a real system, you should always validate on the server.
